# Nice TODO

Example use cases

- Family: persons as columns, tasks in columns, manage weekend tasks
- Work: title Current Week, projects as columns, tasks in columns, at end of week: export as markdown

## Tech Stack Variant 1 (implemented)

- Python 3.11, NiceGUI
- SQLModel + SQLite for persistence
- implemented in `src/`
- unit tests in pytest not needed yet
- run via `python -m src.main`, open `http://localhost:8505/?key=default`

### Architecture

Layered architecture within a single NiceGUI process:

```text
Browser (UI) ←WebSocket→ NiceGUI UI Layer → Service Layer → Repository Layer → sqlite.db
```

- Abstract `BoardRepository` interface allows swapping SQLite for MySQL without changing service/UI code
- `SqliteBoardRepository` uses SQLModel (SQLAlchemy + Pydantic) for ORM
- `BoardService` orchestrates business logic
- `ExportService` generates Markdown exports
- UI components: `CardComponent` (draggable), `ColumnComponent` (droppable), dialogs

## Data Model

### Board

- ID: int, primary key, auto-increment
- Key: string, unique, URL-safe characters only (alphanumeric, hyphen, underscore, tilde, period)
- Name: string
- last_login: ISO 8601 datetime string, updated on each board access

### Column

- ID: int, primary key, auto-increment
- board_id: int, foreign key → Board.ID, CASCADE on delete
- Name: string
- Position: int (0-based, sequential)

### Card

- ID: int, primary key, auto-increment
- column_id: int, foreign key → Column.ID, CASCADE on delete
- Title: string
- Position: int (0-based, sequential within column)
- is_completed: bool (default false)
- is_template: bool (default false)
- label_id: int | null, foreign key → Label.ID, SET NULL on delete

### Label

- ID: int, primary key, auto-increment
- board_id: int, foreign key → Board.ID, CASCADE on delete
- Name: string
- Color: string (hex color, e.g. "#ff5722")

### SQLite Schema Notes

- Table `column_` (underscore suffix because `column` is a SQL reserved word)
- `is_completed` / `is_template` stored as INTEGER (0/1) in SQLite
- Foreign keys enforced via `PRAGMA foreign_keys = ON`

## Features V1 (implemented)

### Board Interactions

All board-level actions are in a dropdown menu (three-dot icon next to board name):

- Add Column
- Manage Labels (create, edit name/color, delete)
- Bulk Label Mode (select multiple cards via checkboxes, assign label to all at once)
- Export All (Markdown: board name as h1, columns as h2, card titles as bullets, empty columns omitted)
- Export Completed (same structure, only completed cards, columns without completed cards omitted)
- Delete Finished Cards (removes cards where is_completed=true AND is_template=false)
- Delete All Cards (removes cards where is_template=false, templates survive)
- Rename Board
- Edit Board Key (with validation: non-empty, URL-safe chars, unique across boards)
- Delete Board: admin-only via CLI script `python -m src.delete_board <id_or_key>`, not exposed in UI

### Column Interactions

- Rename: click column name to edit inline, saves on Enter or blur
- Drag-and-drop reorder: drag handle icon on column header
- Delete: button in column header, with confirmation dialog
- Card count badge displayed next to column name

### Card Interactions

- Create: type title in input at bottom of column, press Enter. Input stays focused for rapid multi-card creation
- Complete/incomplete: green checkbox on each card
- Edit title: click title to edit inline, saves on Enter or blur
- Delete: small X button on card (with low opacity, unobtrusive)
- Assign label: small circle icon on card opens dropdown with all board labels + "Remove label" option
- Move between columns: HTML5 drag-and-drop
- Reorder within column: HTML5 drag-and-drop
- Touch support: polyfill translates touch events to drag events for mobile

### Card Visual Design

- Cards with a label use the exact label color as background
- Completed cards with a label: same color but reduced opacity (0.45)
- Completed cards without a label: light grey background, reduced opacity (0.6)
- Completed card titles: strikethrough + grey text
- Template cards: dashed border + template icon
- Cards have rounded corners (6px), smooth transitions

### Label Features

- Managed per board (board-scoped)
- Each label has a name and a hex color
- Max 1 label per card
- Bulk label mode: enter mode via menu, checkboxes appear on all cards, pick a label to assign to all selected cards. Label buttons in bulk mode use the label's color as background
- Deleting a label clears it from all cards that had it assigned

### UI/UX Design

- Page background: purple-blue gradient (fixed, doesn't scroll)
- Board title: white, bold, on gradient background
- Columns: light grey rounded containers (#eceff1), 280-320px wide
- Cards: white or label-colored, compact (32px min-height), 3px gap between cards
- Horizontal scroll for columns on mobile
- Touch-friendly: all interactive elements have adequate tap targets
- Responsive layout adapts to mobile screen widths

### Markdown Export Format

```markdown
# Board Name

## Column Name
- Card title 1
- Card title 2

## Another Column
- Card title 3
```

- Empty columns are omitted
- "Export Completed" includes only cards with is_completed=true
- Columns with no matching cards are omitted
- Presented in a dialog with copy-to-clipboard button

### Concurrency

- Multiple users can access and modify the same board concurrently
- Last-write-wins: no locking, no conflict detection
- No real-time sync between clients (changes visible on next page load/refresh)

### Authentication

- Token-based via URL query parameter, handled by `.htaccess` on hosting (not by the app)
- App only validates the `key` parameter to identify which board to load
- URL format: `http://host:port/?key=<board_key>`

## Simplifications V1

- No real-time push between concurrent users
- Token auth handled externally (`.htaccess`), not by the app

## Tech Stack

- Web-Hosted on Uberspace
- Optimized for Mobile Devices
- Allow for parallel users to work on same board, on conflicts: the last wins

## Login/Security

- Anonymous users, no registering
- NOT: Anonymous Login via hard-coded token as URL query parameter (same for all users), as that would not secure the Backend
- Login: URL query parameters: `key` (board identifier)
- No permissions system
- No sensitive data in app
